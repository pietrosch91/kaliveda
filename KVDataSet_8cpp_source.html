32 <h2>KVDataSet</h2>
33 <h4>Describes a set of experimental data which may be analysed with KaliVeda</h4>
34 <p>A dataset brings together all the elements necessary for the
35 analysis of data taken during an experiment or campaign of experiments
36 using the same experimental configuration. These are:
37 </p>
38 <ul>
39  <li>the files containing the data in its raw state</li>
40  <li>files containing partially and/or completely calibrated data</li>
41  <li>a description of the experimental set-up (detectors, geometry, etc.)</li>
42  <li>a database describing the different runs &amp; reaction systems
43 studied, detector parameters (status, calibration parameters, etc.)</li>
44 </ul>
45 
46 <h3><a name="init"></a>I. Initialisation</h3>
47 
48 <p>Different datasets are listed in the configuration file <span
49  style="font-family: monospace;">$KVROOT/KVFiles/.kvrootrc</span>. This file is read by
50  <a style="font-family: monospace;" href="KVDataSetManager.html#KVDataSetManager:Init">KVDataSetManager::Init()</a>,
51 which sets up and handles all the datasets for a given data repository
52 (see <a href="KVDataRepository.html"><span
53  style="font-family: monospace;">KVDataRepository</span></a>).
54 The syntax for adding a dataset to this list is described in the file.
55 </p>
56 <p>After creation, the availability of this dataset for analysis is
57 decided based on whether data files associated with it are present or
58 not in the data repository, and, optionally, whether the user's name
59 appears in the associated user group (see <a
60  style="font-family: monospace;" href="#KVDataSet:CheckAvailable">KVDataSet::CheckAvailable()</a>).
61 Datasets which are available for analysis by the user have <span
62  style="font-family: monospace;"><a href="#KVDataSet:IsAvailable">IsAvailable()</a>==kTRUE</span>.
63 </p>
64 <p>All possible <a href="KVDataAnalysisTask.html">data analysis tasks</a>
65 for this dataset are then determined based on the available types of
66 data in the repository (see <a style="font-family: monospace;"
67  href="#KVDataSet:SetAnalysisTasks">SetAnalysisTasks()</a>) and the
68 default list of known analysis tasks (see <a
69  style="font-family: monospace;"
70  href="KVDataSetManager.html#KVDataSetManager:ReadTaskList">KVDataSetManager::ReadTaskList()</a>).
71 Data analysis tasks can be chosen &amp; performed for any available
72 dataset by using a <a style="font-family: monospace;"
73  href="KVDataAnalyser.html">KVDataAnalyser</a>.
74 </p>
75 
76 <h3><a name="what"></a>II. What's In A Dataset ?</h3>
77 
78 <h4>a. Experimental set-up: multidetector array</h4>
79 <p>The experimental set-up for each dataset is handled by a child class
80 of <a style="font-family: monospace;" href="KVMultiDetArray.html">KVMultiDetArray</a>,
81 which can describe anything from a single detector
82 to a 4&#960; (pi)-multidetector of charged particles such as INDRA. Each
83 dataset is uniquely associated to its experimental set-up by defining a
84 <a href="http://root.cern.ch/root/html/TPluginManager.html">plugin</a>
85 of the <span style="font-family: monospace;">KVMultiDetArray</span>
86 class:
87 </p>
88 <pre>+Plugin.KVMultiDetArray: name_of_dataset name_of_class name_of_plugin_library constructor_to_call</pre>
89 <p>See file <span style="font-family: monospace;">$KVROOT/KVFiles/.kvrootrc</span>
90 for examples concerning the default datasets.
91 There you will also see that the multidetector class can be further
92 "tweaked" by defining dataset-dependent plugins for the <a
93  href="KVIDTelescope.html"><span style="font-family: monospace;">KVIDTelescope</span></a>
94 class (see <a
95  href="KVMultiDetArray.html#KVMultiDetArray:GetIDTelescopes"><span
96  style="font-family: monospace;">KVMultiDetArray::GetIDTelescopes</span></a>).
97 </p>
98 <p>The experimental set-up for the dataset can be created when
99 necessary by using <a href="#KVDataSet:BuildMultiDetector"><span
100  style="font-family: monospace;">KVDataSet::BuildMultiDetector()</span></a>.
101 </p>
102 
103 <h4>b. Runs, systems and calibration parameter database</h4>
104 
105 <p>A child class of <a href="KVDataBase.html"><span
106  style="font-family: monospace;">KVDataBase</span></a> handles all
107 stored information on the different experimental runs, reaction systems
108 studied, run-dependent detector characteristics, calibration
109 parameters, etc. The link between dataset and database is defined by a <a
110  href="http://root.cern.ch/root/html/TPluginManager.html">plugin</a>:
111 </p>
112 
113 <pre>+Plugin.KVDataBase: name_of_dataset name_of_class name_of_plugin_library constructor_to_call</pre>
114 
115 <p>See file <span style="font-family: monospace;">$KVROOT/KVFiles/.kvrootrc</span>
116 for examples concerning the default datasets.</p>
117 
118 <p>Such databases are constructed from a number of different source
119 files (runlist, system list, etc.) which are kept in the directory <span
120  style="font-family: monospace;">$KVROOT/KVFiles/name_of_dataset</span>
121 (string returned by <a style="font-family: monospace;"
122  href="#KVDataSet:GetDataSetDir">GetDataSetDir()</a>).
123 The database object, once constructed from these files (see <a
124  href="KVDataBase.html#KVDataBase:Build"><span
125  style="font-family: monospace;">KVDataBase::Build()</span></a>), is
126 saved in a ROOT file in the same directory. The default name for this
127 file is given by the environment variable
128 </p>
129 <pre>DataSet.DatabaseFile: DataBase.root</pre>
130 <p>in <span style="font-family: monospace;">$KVROOT/KVFiles/.kvrootrc</span>.
131 There you will also see examples of definitions of dataset-specific
132 source file names and formats.
133 </p>
134 <p>To acces the database associated with the dataset, use <a
135  style="font-family: monospace;" href="#KVDataSet:GetDataBase">GetDataBase()</a>.<br>
136 </p>
137 
138 <h4>c. Data files</h4>
139 
140 <p>All files of experimental data belonging to this dataset are kept
141 under the <span style="font-family: monospace;">data_repository_subdirectory</span>
142 defined in the configuration of the dataset. The name of this
143 directory is returned by <a style="font-family: monospace;"
144  href="#KVDataSet:GetDatapathSubdir">GetDatapathSubdir()</a>. There are
145 four different types of data, each of which is kept in a different
146 subdirectory of the dataset's repository directory:
147 </p>
148 <ul>
149  <li><span style="font-family: monospace;">raw/</span>&nbsp;&nbsp;&nbsp;
150 The raw data written during the experiment by the experimental
151 acquisition system
152  </li>
153  <li><span style="font-family: monospace;">recon/</span>&nbsp;&nbsp;&nbsp;
154 Reconstructed data in which the (uncalibrated) energy losses measured
155 in different detectors of the set-up have been associated into
156 (unidentified) particles.
157  </li>
158  <li><span style="font-family: monospace;">ident/</span>&nbsp;&nbsp;&nbsp;
159 Reconstructed data of the previous type to which some or all of the
160 necessary identification and calibration algorithms have been applied
161 in order to identify and calibrate the previously reconstructed
162 particles.
163  </li>
164  <li><span style="font-family: monospace;">root/</span>&nbsp;&nbsp;&nbsp;
165 Fully-identified and calibrated reconstructed events resulting from the
166 application of coherency algorithms to the last-produced <span
167  style="font-family: monospace;">ident</span> data files.<br>
168  </li>
169 </ul>
170 <p>It should be noted that the last 3 types of data are obligatorily
171 stored in <a href="http://root.cern.ch/root/html/TTree.html">TTrees</a>
172 written in ROOT files. They also have exactly the same structure, i.e.
173 they are constituted of <a href="KVReconstructedEvent.html">events</a>
174 containing one or more <a href="KVReconstructedNucleus.html">particles</a>.
175 The algorithm for reconstructing raw data, i.e. for creating <span
176  style="font-family: monospace;">recon</span> files from <span
177  style="font-family: monospace;">raw</span> files, is described in <a
178  href="KVReconstructedEvent.html#KVReconstructedEvent:ReconstructEvent"><span
179  style="font-family: monospace;">KVReconstructedEvent::ReconstructEvent</span></a>.
180 Both this process, and also <span style="font-family: monospace;">recon</span>&#8594;<span
181  style="font-family: monospace;">ident</span>, <span
182  style="font-family: monospace;">ident</span>&#8594;<span
183  style="font-family: monospace;">root</span> are among the data
184 analysis tasks predefined (for INDRA data) in <span
185  style="font-family: monospace;">$KVROOT/KVFiles/.kvrootrc</span>.
186 </p>
187 
188 <p>This classification of data is meant to reflect the chronology of a
189 typical process of data reduction &amp; analysis. The first step is to
190 create <span style="font-family: monospace;">recon</span> files from
191 the raw data using the <span style="font-family: monospace;">raw</span>&#8594;<span
192  style="font-family: monospace;">recon</span> analysis task. This can
193 be performed as soon as the description of the experimental set-up (<a
194  href="KVMultiDetArray.html"><span style="font-family: monospace;">KVMultiDetArray</span></a>
195 class) has been implemented and the bare minimum of source files for
196 setting up the database (runlist) are available. The advantage of
197 working with recon files is that (i) the event reconstruction is
198 performed once, in the <span style="font-family: monospace;">raw</span>&#8594;<span
199  style="font-family: monospace;">recon </span>step, so subsequent
200 analysis is that much faster; (ii) the data is stored in TTrees in ROOT
201 files which, unless the acquisition system is based on ROOT, means that
202 they can be analysed much more efficiently and on more platforms than
203 the original raw files.
204 </p>

561 void KVDataSet::SaveDataBase() const
562 {
563  // Write the database to disk (ROOT file).
564  // It will be written in the directory
565  // $KVROOT/db/[dataset name]
566  // If the directory does not exist, it will be created. Permissions are set to 775 (rwxrwxr-x).
567  //
568  // # Default name of database file containing informations on runs, systems, calibration parameters etc.
569  // DataSet.DatabaseFile: DataBase.root
570  // # Default name of database object in file
571  // DataSet.DatabaseName: DataBase
572 
573  TString dbfile_fullpath = GetFullPathToDB();
574  TString tmp = gSystem->DirName(dbfile_fullpath.Data()); //full path to directory $KVROOT/db/[dataset name]
575 
576  if (gSystem->AccessPathName(tmp.Data())) { // directory $KVROOT/db/[dataset name] does not exist
577 
578  if (gSystem->mkdir(tmp.Data()) == -1) { // problem creating $KVROOT/db/[dataset name]
579 
580  TString tmp2 = gSystem->DirName(tmp.Data());// full path to directory $KVROOT/db
581 
582  if (gSystem->AccessPathName(tmp2.Data())) { // directory $KVROOT/db does not exist
583 
584  if (gSystem->mkdir(tmp2.Data()) == -1) { // problem creating $KVROOT/db
585  Error("SaveDataBase", "Cannot create directory %s required to save database",
586  tmp2.Data());
587  return;
588  }
589  gSystem->Chmod(tmp2.Data(), 0775);
590  } else {
591  Error("SaveDataBase", "Cannot create directory %s required to save database, even though %s exists: check disk space ?",
592  tmp.Data(), tmp2.Data());
593  return;
594  }
595  //try again
596  if (gSystem->mkdir(tmp.Data()) == -1) {
597  Error("SaveDataBase", "Cannot create directory %s required to save database",
598  tmp.Data());
599  return;
600  } else {
601  gSystem->Chmod(tmp.Data(), 0775);
602  }
603  } else {
604  gSystem->Chmod(tmp.Data(), 0775);
605  }
606  }
607 
608  WriteDBFile(dbfile_fullpath.Data());
609 }

702 void KVDataSet::OpenDataBase(Option_t* opt) const
703 {
704  //Open the database for this dataset.
705  //If the database does not exist or is older than the source files
706  //the database is automatically rebuilt
707  //(see DataBaseNeedUpdate()).
708  //Use opt="UPDATE" to force rebuilding of the database.
709  //
710  //First, we look in to see if the database file exists
711  //(if no database file name given, use default name for database file defined in
712  //.rootrc config files).
713  //If so, we open the database contained in the file, after first loading the required plugin
714  //library if needed.
715  //
716  //The name of the dataset must correspond to the name of one of the Plugin.KVDataBase
717  //plugins defined in the .rootrc configuration files
718  //
719  //WARNING: if the database needs to be (re)built, we set gDataSet to
720  //point to this dataset in case it was not already done,
721  //as in order to (re)build the database it may be necessary for
722  //gDataSet to point to the current dataset.
723 
724  Bool_t is_glob_db = kFALSE;
725  //if option="update" or database out of date or does not exist, (re)build the database
726  if ((!strcmp(opt, "UPDATE")) || DataBaseNeedsUpdate()) {
727  //check if it is the currently active database (gDataBase),
728  //in which case we must 'cd()' to it after rebuilding
729  Info("OpenDataBase", "Updating database file");
730  is_glob_db = (fDataBase == gDataBase);
731  if (fDataBase) {
732  delete fDataBase;
733  fDataBase = 0;
734  }
735  if (fDBase) {
736  delete fDBase;
737  fDBase = 0;
738  }
739  // make sure gDataSet is set & points to us
740  gDataSet = const_cast<KVDataSet*>(this);
741  fDataBase = KVDataBase::MakeDataBase(GetDBName(), GetDataSetDir());
742  if (!fDataBase) {
743  // no database defined for dataset
744  Info("OpenDataBase", "No database defined for dataset");
745  return;
746  }
747  SaveDataBase();
748  if (fDataBase && is_glob_db) fDataBase->cd();
749  } else if (!fDataBase) {
750  // if database is not in memory at this point, we need to
751  // open the database file and read in the database
752 
753  //load plugin for database
754  if (!LoadPlugin("KVDataBase", GetName())) {
755  Error("GetDataBase", "Cannot load required plugin library");
756  return;
757  }
758  //look for database file in dataset subdirectory
759  TString dbfile_fullpath = GetFullPathToDB();
760  //open database file
761  OpenDBFile(dbfile_fullpath.Data());
762  }
763 }

839 void KVDataSet::CheckAvailable()
840 {
841  //Check if this data set is physically present and available for analysis.
842  //In other words we check if the value of GetDatapathSubdir() is a subdirectory
843  //of the given data repository
844  //If so, we proceed to check for the existence of any of the datatypes defined in
845  //
846  //KVDataSet.DataTypes:
847  //
848  //by checking for the associated sudirectories defined in the corresponding variables:
849  //
850  //KVDataSet.DataType.Subdir.[type]:
851  //
852  //If none of them exists, the dataset will be reset to 'unavailable'
853  //Otherwise the corresponding flags are set.
854  //
855  //Note that if SetUserGroups has been called with a list of groups allowed to read this data,
856  //the current user's name (gSystem->GetUserInfo()->fUser) will be used to check if the
857  //dataset is available. The user name must appear in the group defined by SetUserGroups.
858 
859  if (!fRepository) // for a stand-alone KVDataSetManager not linked to a KVDataRepository,
860  SetAvailable(); // all known datasets are 'available'
861  else
862  SetAvailable(fRepository->CheckSubdirExists(GetDataPathSubdir()));
863  if (!IsAvailable())
864  return;
865  //check subdirectories
