31 <h2>KVDataAnalyser</h2>
32 <h4>Set up and run data analysis tasks</h4>
33 <p>
34 This can be used either 'interactively' or in menu-driven mode.
35 In both cases, in order to run a data analysis task, the user must
36 specify:
37 </p>
38 <ul>
39 <li>the dataset to analyse</li>
40 <li>the type of analysis task to perform</li>
41 <li>the run or runs to analyse</li>
42 <li>whether to use a batch system to execute the analysis</li>
43 </ul>
44 <p>
45 The role of this base class is to set up the environment necessary for the execution of
46 a given data analysis task;
47 the actual analysis will be performed by a specialised child class of KVDataAnalyser as
48 specified by the task. See <a href="KVDataAnalysisTask.html">KVDataAnalysisTask</a>.
49 The available data analysers are defined as plugins in the file $KVROOT/KVFiles/.kvrootrc
50 as in the following example:
51 </p>
52 <pre>
53 # Plugins for data analysis
54 # KVDataAnalyser can be tuned for different datasets/environments
55 Plugin.KVDataAnalyser: INDRA KVINDRADataAnalyser KVIndra "KVINDRADataAnalyser()"
56 +Plugin.KVDataAnalyser: INDRAReconData KVINDRAReconDataAnalyser KVIndra "KVINDRAReconDataAnalyser()"
57 +Plugin.KVDataAnalyser: INDRARawDataRecon KVINDRARawDataReconstructor KVIndra "KVINDRARawDataReconstructor()"
58 +Plugin.KVDataAnalyser: INDRAGeneDataAnalyser KVINDRAGeneDataAnalyser KVIndra "KVINDRAGeneDataAnalyser()"
59 </pre>
60 <h4>Perform analysis tasks using text menus to choose data, task, runs, etc.</h4>
61 
62 <pre>
63 KVDataAnalyser* datan = new KVDataAnalyser;
64 datan->RunMenus()
65 </pre>
66  --- or ---
67 <pre>
68 gDataSetManager-><a href="KVDataSetManager.html#KVDataSetManager:RunAnalyser">RunAnalyser(...)</a>
69 </pre>
70 
71 <h4>Perform analysis tasks "interactively"</h4>
72 
73 <p>In an interactive ROOT session, type:</p>
74 
75 <pre>
76 KVDataAnalyser* datan = new KVDataAnalyser;
77 datan->SetDataSet(...)
78 datan->SetAnalysisTask(...)
79 datan->SetRuns(...)
80 etc. etc.
81 </pre>
82 
83 <p>to set up the analysis task. You can also mix 'interactive' and menus by calling</p>
84 
85 <pre>
86 datan->ChooseDataSet()
87 datan->ChooseAnalysisTask()
88 datan->ChooseSystem()
89 datan->ChooseRuns(system)
90 </pre>
91 
92 <p>Note that in this case the order is important: you cannot "ChooseAnalysisTask"
93 before choosing or setting a dataset. Note also that in order to use ChooseRuns()
94 you must give the pointer to the system (KVDBSystem*) to analyse.</p>
95 
96 <p>To set the "running mode", i.e. whether to run in batch or not, you can use</p>
97 <pre>
98 datan->ChooseRunningMode()
99 </pre>
100 <p>to be asked to choose interactive or batch, and then (if required) a batch system.</p>
101 <p>If you don't use ChooseRunningMode() then use either:</p>
102 <pre>
103 datan->SetBatchSystem(0); // no batch
104 </pre>
105 <p>or, if you want to use a batch system, you can get a pointer to the required batch system
106 from the <a href="KVBatchSystemManager.html">batch system manager</a>:</p>
107 <pre>
108 KVBatchSystem *mybatch = gBatchSystemManager->GetBatchSystem(...)
109 </pre>
110 <p>define the batch properties (see <a href="KVBatchSystem.html">KVBatchSystem</a> and derived classes),</p>
111 <pre>
112 mybatch->SetJobName(...)
113 mybatch->SetMultiJobsMode(); // to run 1 job for each run
114 </pre>
115 <p>and then do</p>
116 <pre>
117 datan->SetBatchSystem( mybatch ); // run task in batch
118 </pre>
119 <p>When ready to perform the analysis, call</p>
120 <pre>
121 datan->Run()
122 </pre>
123 <p>You will be asked to supply any missing parameters for the analysis task.</p>
124 
125 <h4>Naming user files generated by analysis in batch mode</h4>
126 
127 <p>When running an analysis task using a batch system, the working directory where the
128 analysis is executed may be different to the one where the job was submitted. The source files
129 for any user-supplied analysis class are automatically copied to the working directory when
130 execution of the task begins. <b>Any</b> files generated by the user's analysis are copied back to the original directory at the end of
131 execution. In case several tasks are launched from the same directory, it is the user's responsibility
132 to give unique names to the files generated by her analysis. One simple way to do this is to
133 use the name of the batch job as a basis for naming such files (see <a href="KVBatchSystem.html">KVBatchSystem</a>):
134 </p>
135 <pre>
136 #include "KVBatchSystem.h"
137 ...
138 if( gBatchSystem ) {
139  //running in batch. name results file after job name.
140  new TFile( Form( "%s.root", gBatchSystem->GetJobName() ), "recreate");
141 }
142 else {
143  //not running in batch. fixed name.
144  new TFile("Results.root", "recreate");
145 }
146 </pre>

349 void KVDataAnalyser::Run()
350 {
351  //Check all task variables, then run analyser
352 
353  fSubmit = kFALSE;
354  //print welcome message for main (parent) analyser
355  if (!fParent && !fMenus) KVBase::PrintSplashScreen();
356 
357  if (CheckTaskVariables()) {
358  if (fBatchSystem && !BatchMode()) {
359  //if batch mode is requested, the job is submitted to the chosen batch system
360  fBatchSystem->SubmitTask(this);
361  } else {
362  if (BatchMode() && fBatchSystem && !fParent) fBatchSystem->Print("log");
363  if (!RunningInLaunchDirectory() && fParent) {
364  //when batch job runs in directory different to launch directory,
365  //we scan the list of files present in the current working directory
366  //just prior to running the analysis task
367  ScanWorkingDirectory(&fWorkDirInit);
368  }
369  if (!PreSubmitCheck()) return;
370 
371  // The following horrific kludge is supposed to solve the following problem:
372  // when running in batch mode (GridEngine) the data analyser created in
373  // KaliVedaAnalysis.cpp has to be of the right type (KVSimDirAnalyser or KVDataSetAnalyser)
374  // in order for it to read all necessary informations from the batch env file
375  // by calling the appropriate ReadBatchEnvFile override.
376  // However, when Run() is called for this object from KaliVedaAnalysis.cpp,
377  // we have to call the KVDataAnalyser::SubmitTask method, not the override in the
378  // derived class.
379  if (fUseBaseClassSubmitTask)
380  KVDataAnalyser::SubmitTask();
381  else
382  SubmitTask();
383 
384  if (!RunningInLaunchDirectory() && fParent) {
385  //when batch job runs in directory different to launch directory,
386  //we scan the list of files present in the current working directory
387  //just after running the analysis task
388  ScanWorkingDirectory(&fWorkDirEnd);
389  //any files which are present in the second list which were not present
390  //in the first list will be copied back to the launch directory
391  CopyAnalysisResultsToLaunchDirectory();
392  }
393  if (BatchMode() && fBatchSystem && !fParent) {
394  // at end of batch jobs,
395  // remove .[jobname] and [jobname].status files from $HOME directory
396  // remove .[jobname].bak and [jobname].status.bak files from $HOME directory
397  TString ff;
398  AssignAndDelete(ff, gSystem->ConcatFileName(gSystem->Getenv("HOME"), Form(".%s", GetBatchName())));
399  gSystem->Unlink(ff);
400  AssignAndDelete(ff, gSystem->ConcatFileName(gSystem->Getenv("HOME"), Form(".%s.bak", GetBatchName())));
401  gSystem->Unlink(ff);
402  AssignAndDelete(ff, gSystem->ConcatFileName(gSystem->Getenv("HOME"), Form("%s.status", GetBatchName())));
403  gSystem->Unlink(ff);
404  AssignAndDelete(ff, gSystem->ConcatFileName(gSystem->Getenv("HOME"), Form("%s.status.bak", GetBatchName())));
405  gSystem->Unlink(ff);
406  }
407  }
408  }
409  PostRunReset();
410 }

525 void KVDataAnalyser::SetUserIncludes(const Char_t* incDirs)
526 {
527 // Add to the includes paths the user's includes paths
528 // the includes paths have to be separated by a white space
529 
530  fIncludes = "";
531  if (!incDirs) {
532  return;
533  }
534  TString tmp = incDirs;
535  TString curIncDir = gSystem->GetIncludePath();
536  TObjArray* oa = tmp.Tokenize(" ");
537  oa->SetOwner(kTRUE);
538  TIter next(oa);
539  TObjString* st = 0;
540  while ((st = (TObjString*)next())) {
541  TString id = st->GetString();
542  if (id.Length()) {
543  fIncludes += id.Data();
544  fIncludes += " ";
545  if (!curIncDir.Contains(id.Data())) {
546  cout << "Include path \"" << id.Data() << "\" added." << endl;
547  id.Prepend("-I");
548  gSystem->AddIncludePath(id.Data());
549  }
550  }
551  }
552  delete oa;
553 }

701 Bool_t KVDataAnalyser::CheckIfUserClassIsValid(const KVString& alternative_base_class)
702 {
703  //Return kTRUE if the name of the class given by the user (fUserClass) is valid
704  //for the analysis task. This is so if one of the following is true:
705  // - the class library has already been loaded. In this case the class will exist
706  // in the dictionary (gROOT->GetClass()); we check if it derived from the
707  // base class defined for the analysis task
708  // - a plugin exists defining this class as an extension of the base class defined
709  // for the analysis task (gROOT->GetPluginManager()->FindHandler(...): the URI for
710  // the plugin must be the same as the name of the class)
711  // - source files for the class are present in the working directory. In this case
712  // we can add a plugin handler for the class.
713  //In the latter two cases, the class is valid if compilation succeeds.
714  //
715  //If the user's class may in fact be derived from an alternative base class, rather
716  //than the base class defined for this analysis task (see KVDataAnalysisTask::SetUserBaseClass)
717  //you can supply the name of this class.
718 
719  TObject* o = GetInstanceOfUserClass(alternative_base_class);
720  if (o) {
721  delete o;
722  return kTRUE;
723  }
724  return kFALSE;
725 }

741 const Char_t* KVDataAnalyser::GetACliCMode()
742 {
743  // Returns string to be appended to name of user class for compilation with ACliC in
744  // GetInstanceOfUserClass. This depends on the boolean resources:
745  //
746  // KVDataAnalyser.UserClass.Debug: ( "yes" => "g" )
747  // KVDataAnalyser.UserClass.Optimise: ( "yes" => "O" )
748  // KVDataAnalyser.UserClass.ForceRecompile: ( "no" => "+"; "yes" => "++" )
749  //
750  // Note that if both Debug and Optimise are set to "yes/true", we use Debug mode.
751  // (can't have BOTH debug & optimisation).
752 
753  static TString aclic;
754  if (gEnv->GetValue("KVDataAnalyser.UserClass.ForceRecompile", kFALSE)) aclic = "++";
755  else aclic = "+";
756  if (gEnv->GetValue("KVDataAnalyser.UserClass.Debug", kFALSE)) aclic += "g";
757  else if (gEnv->GetValue("KVDataAnalyser.UserClass.Optimise", kFALSE)) aclic += "O";
758  return aclic.Data();
759 }

789 TObject* KVDataAnalyser::GetInstanceOfUserClass(const KVString& alternative_base_class)
790 {
791  //Return an instance of the class given by the user (fUserClass), if it is valid.
792  //If the user class is given in the form of source code, it will be (re)compiled
793  //if it has not already been loaded and/or the source has changed since the last
794  //build, using ACliC. If the resource
795  //
796  // KVDataAnalyser.UserClass.Debug: yes
797  //
798  //is set, the user's class will be compiled with extra debugging information
799  //
800  //Once compiled, we check that the user's class is indeed derived from the base
801  //class defined for this analysis task (see KVDataAnalysisTask::SetUserBaseClass).
802  //If the user's class may in fact be derived from an alternative base class, you
803  //can supply the name of this class.
804 
805  // make sure any required plugin library defining base class for user's analysis class is loaded
806  if (!fTask->CheckUserBaseClassIsLoaded()) return 0x0;
807 
808  //do we have a plugin ?
809  TPluginHandler* ph = gROOT->GetPluginManager()->FindHandler(fTask->GetUserBaseClass(), fUserClass.Data());
810  if (!ph) {//no plugin defined
811 
812  //if it is a precompiled class (i.e. already part of KaliVeda),
813  //it will be in the dictionary already
814  TClass* cl = gROOT->GetClass(fUserClass.Data());
815 
816  //do we have source files ?
817  if (DoUserClassFilesExist()) {
818  //compile & load user's source files using ACLIC. ACliC options read by GetACliCMode() from .kvrootrc
819  TString cmd;
820  cmd.Form(".L %s%s", fUserClassImp.Data(), GetACliCMode());
821  gROOT->ProcessLine(cmd.Data());
822  //class will be in dictionary if compilation successful
823  cl = gROOT->GetClass(fUserClass.Data());
824  } else if (!cl) {
825  //class not in dictionary and no source files. help!
826  Info("GetInstanceOfUserClass", "Class %s is unknown and no source files available",
827  fUserClass.Data());
828  return 0;
829  }
830  if (!cl) {
831  //compilation of user class has failed
832  Info("GetInstanceOfUserClass", "Compilation of class %s failed. Correct the mistakes and try again",
833  fUserClass.Data());
834  return 0;
835  }
836  if (!cl->GetBaseClass(fTask->GetUserBaseClass())) {
837  //class does not inherit from correct base
838  if (alternative_base_class == "" || !cl->GetBaseClass(alternative_base_class)) {
839  if (alternative_base_class != "")
840  Info("GetInstanceOfUserClass", "Class %s does not inherit from correct base class (%s or %s), or compilation of class %s failed. Correct the mistakes and try again",
841  fUserClass.Data(), fTask->GetUserBaseClass(), alternative_base_class.Data(), fUserClass.Data());
842  else
843  Info("GetInstanceOfUserClass", "Class %s does not inherit from correct base class (%s), or compilation of class %s failed. Correct the mistakes and try again",
844  fUserClass.Data(), fTask->GetUserBaseClass(), fUserClass.Data());
845  return nullptr;
846  }
847  }
848  //EVERYTHING OK!! now instanciate an object of the new class
849  return (TObject*)cl->New();
850  } else {
851  Info("GetInstanceOfUserClass", "Found plugin handler for class %s",
852  fUserClass.Data());
853  //load class from plugin
854  ph = KVBase::LoadPlugin(fTask->GetUserBaseClass(), fUserClass.Data());
855  if (!ph) {
856  Info("GetInstanceOfUserClass", "KVBase::LoadPlugin failed for %s", fUserClass.Data());
857  return 0;
858  }
859  TObject* obj = (TObject*)ph->ExecPlugin(0);
860  if (obj) {
861  //Info("GetInstanceOfUserClass", "constructor OK for %s", fUserClass.Data());
862  if (obj->InheritsFrom(fTask->GetUserBaseClass())) return obj;
863  Info("GetInstanceOfUserClass", "%s does not inherit from %s", fUserClass.Data(), fTask->GetUserBaseClass());
864  return 0;
865  } else {
866  Info("GetInstanceOfUserClass", "constructor not OK for %s", fUserClass.Data());
867  return 0;
868  }
869  }
870  return 0;
871 }

1087  //the user-provided class is to be used as analyser
1088  the_analyser.reset((KVDataAnalyser*)GetInstanceOfUserClass());

1090  the_analyser.reset(GetAnalyser(fTask->GetDataAnalyser()));

1100  } else if (strcmp(fTask->GetUserBaseClass(), "")) the_analyser->SetUserClass(fTask->GetUserBaseClass(), kFALSE);
1101  if (!BatchMode()) {
1102  //when not in batch mode i.e. when submitting a task, we ask the user to supply
1103  //any further information required by the task, and then ask whether to run in
1104  //interactive or batch mode
